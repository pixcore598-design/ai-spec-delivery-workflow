# Global Claude Workflow

本机默认协作流程采用 `OpenSpec + Superpowers + gstack` 思路。

## Default Policy

1. **规范先行**
   - 对于中大型功能、跨模块改动、重构、公共接口变更，先给出 `proposal / design / tasks` 三层结构，再开始实现。
   - 如果没有正式 `openspec/` 目录，也必须在回复中给出等价结构化摘要。

2. **歧义先澄清**
   - 需求边界不清、验收标准不明、影响范围不确定时，先提问再编码。
   - 禁止在关键假设未确认时直接大范围修改代码。

3. **小步执行**
   - 任务优先拆为小步，逐步实现、逐步验证。
   - 只读调研、搜索、扫描类工作优先交给子代理或搜索能力处理。

4. **验证优先**
   - 没有测试、运行结果、截图、QA 结果或其他有效证据，不宣告完成。
   - UI 或浏览器相关变更，优先补页面验证证据。

5. **轻重分流**
   - 小任务：简短对齐 + 直接实现 + 定向验证。
   - 中任务：规范摘要 + 澄清 + 分步实现 + 验证。
   - 大任务：完整规范 -> 拆解 -> 实现 -> QA/截图/测试 -> 收尾。

## Skill Routing

- 需要完整规范驱动交付时，优先使用 `spec-delivery-workflow`
- 需要广泛调研、跨模块分析时，优先使用 `agent-reach`
- 需要浏览器操作、截图、页面验证时，优先使用浏览器类能力
- 前端视觉质量要求高时，可结合 `frontend-design`

## Completion Standard

交付结果至少包含：

- 做了什么
- 怎么验证的
- 剩余风险或限制
- 建议的下一步

关键原则：

- 没有证据，不算完成
- 没有对齐，不做大任务
- 没有澄清，不做关键假设跳跃

## Recommended User Prompts

- “按 spec delivery workflow 推进这个任务”
- “先规范拆解，再实现并验证”
- “Treat this as a production-grade delivery task”
- “不要直接开写，先对齐需求和验收标准”
